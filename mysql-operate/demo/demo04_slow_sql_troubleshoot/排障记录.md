# SQL慢查询排障优化记录

## 1. 建表，导入数据。

表结构：
```
CREATE TABLE `tb_sku` (
  `id` int(11) NOT NULL AUTO_INCREMENT COMMENT '商品id',
  `sn` varchar(100) NOT NULL COMMENT '商品条码',
  `name` varchar(200) NOT NULL COMMENT 'SKU名称',
  `price` int(20) NOT NULL COMMENT '价格（分）',
  `num` int(10) NOT NULL COMMENT '库存数量',
  `alert_num` int(11) DEFAULT NULL COMMENT '库存预警数量',
  `image` varchar(200) DEFAULT NULL COMMENT '商品图片',
  `images` varchar(2000) DEFAULT NULL COMMENT '商品图片列表',
  `weight` int(11) DEFAULT NULL COMMENT '重量（克）',
  `create_time` datetime DEFAULT NULL COMMENT '创建时间',
  `update_time` datetime DEFAULT NULL COMMENT '更新时间',
  `category_name` varchar(200) DEFAULT NULL COMMENT '类目名称',
  `brand_name` varchar(100) DEFAULT NULL COMMENT '品牌名称',
  `spec` varchar(200) DEFAULT NULL COMMENT '规格',
  `sale_num` int(11) DEFAULT '0' COMMENT '销量',
  `comment_num` int(11) DEFAULT '0' COMMENT '评论数',
```

## 2. 查询数据，追踪、优化慢查询

慢查询语句1（经日志捕捉到执行时间为1.07s)：查询指定品牌、品类的上架商品的ID、名字和销量

```
SELECT id,name,sale_num FROM tb_sku 
WHERE category_name = '牛仔裤' AND status = '1' AND brand_name='ME&CITY' 
ORDER BY id;

```
原因分析：WHERE 子句中三个筛选字段均无索引，系统只能执行全表扫描，逐行读取完整数据表校验筛选条件，百万级数据下 IO 开销巨大。
优化办法：建立包含category_name, status, brand_name的联合索引。
```
create index idx_sku_cate_sta_bra 
on tb_sku(category_name,status,brand_name);
```
优化效果：
字段|优化前|优化后|性能解读
--|--|--|--
select_type|SIMPLE|SIMPLE|单表普通查询，无关联子查询，无变化
table|tb_sku|tb_sku | 操作同一张商品表
type|index|ref | 巨大提升，优化前 type=index 遍历整棵主键索引树，优化后 ref 等值精准匹配索引
possible_keys|NULL|idx_sku_cate_sta_bra | 优化前无可用业务索引，优化后匹配自建联合索引
key|PRIMARY|idx_sku_cate_sta_bra | 优化前仅能遍历主键索引，优化后命中业务联合索引
key_len|4|1211|key_len 为索引占用字节，联合索引覆盖多筛选字段
ref|NULL|const,const,const | 优化前无常量匹配，优化后 3 个等值条件全部常量精准匹配
rows|1802083|2029 | 预估扫描行数从 180 万骤降至 2029，IO 读取大幅减少
filtered|0.10|100.00 | 优化前仅 0.1% 有效数据，优化后索引直接筛选全部符合条件数据
Extra|Using where|Using index condition | 优化前服务层过滤数据，优化后 ICP 索引下推减少回表 IO
实测执行耗时 |1.07s|0.01s | 整体查询性能大幅提升

验证：
- 优化后重新执行该 SQL，返回结果与优化前一致。
- 使用覆盖索引可进一步避免回表，综合执行优化时间与索引所占空间考虑，本次未使用该优化方法。

慢查询语句2（经日志捕捉到执行时间1.43s）：按种类和品牌名分组计数。
```
select category_name,brand_name,count(*) from tb_sku 
group by category_name,brand_name;
```
原因分析：分组字段顺序不符合语句1优化后创建的索引顺序，导致部分索引失效。
优化办法：建立包含category_name,  brand_name的联合索引。
```
create index idx_sku_cate_bra 
on tb_sku(category_name,brand_name);
```
优化效果：
字段|优化前|优化后|性能解读
--|--|--|--
select_type|SIMPLE|SIMPLE|单表分组查询，无联表、子查询，无变化 table|tb_sku|tb_sku|操作同一张商品SKU表 
type|index|index|均走索引扫描，但索引有序性差异导致处理成本天差地别
key|idx_sku_cate_sta_bra|idx_cate_bra|优化前使用字段顺序与group by不匹配的索引；优化后命中和分组字段完全匹配的联合索引 
key_len|1211|1206|均为长多字段复合索引，优化后索引字段更少，占用空间略小 
ref|NULL|NULL|无等值筛选条件，分组场景无常量索引匹配 
rows|1902083|1902083|预估索引扫描总行数相同，但分组运算开销差距巨大 
filtered|100.00|100.00|全量索引数据参与分组统计，不存在过滤逻辑 
Extra|Using index; Using temporary; Using filesort|Using index|优化前需临时表存储分组数据+文件排序，内存磁盘消耗极高；优化后覆盖索引天然有序，直接完成分组，无额外排序与临时表开销 
实测执行耗时|1.43s|0.40s|消除临时表、文件排序两大性能瓶颈，查询效率大幅提升

验证：

-   优化后重新执行该 SQL，返回结果与优化前一致。
-   针对1、2语句优化的两个索引可合并为字段顺序为category_name,  brand_name,status的联合索引，进一步节省空间。
